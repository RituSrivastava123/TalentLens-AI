## Overview

Hiring teams spend hours reading resumes that don't match the job. TalentLens AI automates the first layer of screening — not to replace recruiters, but to give them a ranked, reasoned shortlist they can act on immediately.

Upload a Job Description. Upload resumes. The agent parses both, extracts requirements, runs semantic matching, scores each candidate across five weighted dimensions, and produces a ranked leaderboard with justifications. Recruiters can override any score before the final PDF report is generated.

The system is built around **explainability** — every score comes with a one-line reasoning so recruiters understand *why* a candidate ranked where they did, not just what their number is.

| **Skill Gap Analysis** | Shows exactly which required skills a candidate is missing |
| **AI Summary** | One-paragraph candidate evaluation generated by LLM reasoning |
| **Analytics Dashboard** | Score distribution charts, heatmaps, and comparative visualizations |

## AI Workflow

### Step-by-Step Process

```
Step 1: Input Ingestion
────────────────────────
Recruiter uploads:
  ├── Job Description (PDF / plain text)
  └── Candidate Resumes (PDF / DOCX, up to N files)

Step 2: JD Parsing
────────────────────────
JD Parser Agent extracts:
  ├── Required Skills        → ["Python", "LangChain", "SQL", ...]
  ├── Experience Requirement → "2+ years in ML engineering"
  ├── Education Requirement  → "B.Tech CS/AIML or equivalent"
  └── Certifications         → ["AWS", "TensorFlow Developer"]

Step 3: Resume Parsing
────────────────────────
Resume Parser Agent processes each file:
  ├── Extracts raw text (pdfplumber / python-docx)
  ├── Identifies sections: Skills, Work Exp, Education, Projects
  └── Normalizes into structured candidate profile

Step 4: Semantic Matching
────────────────────────
For each candidate:
  ├── Embed JD requirements  → vector (all-MiniLM-L6-v2)
  ├── Embed candidate profile → vector
  └── Cosine similarity score → overall match %

Step 5: LLM Scoring
────────────────────────
LLM evaluates each dimension with reasoning:
  ├── Skills Match     → score + justification
  ├── Experience       → score + justification
  ├── Education        → score + justification
  ├── Projects         → score + justification
  └── Communication    → score + justification

Step 6: Weighted Total Calculation
────────────────────────
  Weighted Total =
    (Skills × 0.30) +
    (Experience × 0.25) +
    (Education × 0.15) +
    (Projects × 0.20) +
    (Communication × 0.10)

Step 7: Ranking + Dashboard
────────────────────────
  └── Candidates sorted by weighted total
  └── Skill gap analysis shown per candidate
  └── Visual leaderboard displayed in Streamlit

Step 8: Human Review
────────────────────────
  └── Recruiter can override any score
  └── Override reason is logged
  └── Final scores updated

Step 9: Report Export
────────────────────────
  ├── PDF Report  → shortlist with justifications
  ├── CSV Export  → all scores for HR records
  └── JSON Export → structured data for integrations
```

## Tech Stack & Decision Log

### Component Choices and Why

| Component | Technology | Reason |
|---|---|---|
| **LLM** | `google/flan-t5-base` | Free, runs on CPU, no API cost — ideal for prototype and Colab environment. Instruction-tuned for structured evaluation tasks. |
| **Embeddings** | `all-MiniLM-L6-v2` | Fast semantic similarity with strong performance on short-to-medium text. 384-dim vectors strike the right balance between speed and accuracy. |
| **Resume Parsing** | `pdfplumber` + `python-docx` | Handles both formats reliably. pdfplumber preserves layout better than PyPDF2 for structured resumes. |
| **Agent Framework** | Modular LangChain-inspired pipeline | Gives clear separation between parsing, scoring, and ranking stages without over-engineering for a prototype. |
| **UI** | Streamlit | Fastest path to an interactive recruiter-facing dashboard. Supports file uploads, sliders (human override), and chart rendering out of the box. |
| **Data Layer** | Pandas + SQLite | Lightweight. Pandas for in-session manipulation; SQLite for persistence between runs if needed. |
| **Report Generation** | ReportLab / FPDF | Programmatic PDF generation with full control over layout — better than HTML-to-PDF for structured reports. |
| **Security** | `python-dotenv` + Pydantic | dotenv keeps credentials out of code; Pydantic enforces structured LLM outputs and validates all user inputs. |

### Prompt Design Philosophy

All LLM prompts were designed with four principles:

1. **Fixed evaluation dimensions** — the model is constrained to score only the five specified categories, preventing it from inventing its own criteria.

2. **Structured output enforcement** — prompts instruct the model to return only valid JSON, which is then validated via Pydantic before being used.

3. **Justification requirement** — each score must be accompanied by a one-line reason, making the output auditable and explainable to recruiters.

4. **Conservative defaults** — when the model is uncertain, it defaults to a lower score rather than hallucinating evidence of qualifications.

Example system prompt pattern:

```
You are a professional HR evaluation assistant.

Evaluate the candidate strictly across these dimensions:
  1. Skills Match
  2. Experience Relevance
  3. Education Fit
  4. Project Quality
  5. Communication Clarity

Score each from 0 to 10.
For each dimension, provide a one-sentence justification.

Return ONLY valid JSON in this format:
{
  "skills_score": <int>,
  "skills_justification": "<string>",
  "experience_score": <int>,
  ...
}
```

## Scoring System

### Dimension Weights

```
┌──────────────────────────────────────────────────────┐
│              Candidate Scoring Breakdown             │
├──────────────────┬────────┬────────────────────────  │
│  Dimension       │ Weight │  What's Evaluated        │
├──────────────────┼────────┼────────────────────────  │
│  Skills Match    │  30%   │  Technical skill overlap │
│  Experience      │  25%   │  Relevance + years       │
│  Education       │  15%   │  Degree + institution    │
│  Projects        │  20%   │  Practical AI/tech work  │
│  Communication   │  10%   │  Resume clarity/quality  │
└──────────────────┴────────┴────────────────────────  │
└──────────────────────────────────────────────────────┘

Weighted Total = (S×0.30) + (E×0.25) + (Ed×0.15) + (P×0.20) + (C×0.10)
```

## Security Mitigations

Security is treated as a first-class concern, not an afterthought.

| Risk | Mitigation Applied |
|---|---|
| **API Key Exposure** | All credentials stored in `.env` file, excluded from version control via `.gitignore` |
| **Prompt Injection** | User-uploaded resume text is sanitized before being inserted into prompts. Control characters stripped. |
| **PII Leakage** | Resume data is processed locally and never sent to external servers (model runs on-device) |
| **LLM Hallucinations** | Pydantic schema validation enforces structured output — invalid JSON from the model is rejected and retried |
| **Malicious File Upload** | File type validation on upload (only `.pdf` and `.docx` accepted); file size limits enforced |
| **Unauthorized Access** | Streamlit session state isolates user data per session; no shared state between users |
| **Data Retention** | No persistent storage of candidate data beyond the active session unless explicitly exported by the recruiter |
| **Over-reliance on AI** | Human override layer ensures a human makes the final hiring decision — the AI provides a recommendation, not a verdict |

## Human-in-the-Loop Design

A key design principle of TalentLens AI is that the AI assists recruiters — it does not replace them.

After the automated ranking is complete, the Streamlit dashboard presents each candidate's scores to the recruiter. For any candidate, the recruiter can:

- See the AI's justification for each dimension score
- Manually adjust any score using a slider
- Add a note explaining the override reason
- Re-run the ranking with the adjusted scores

All overrides are logged, creating an audit trail of human decisions. The final PDF report clearly marks which scores were AI-generated and which were manually reviewed.

This design ensures that bias from the AI model — whether from training data or prompt design — can always be caught and corrected by a human before any hiring decision is made.
